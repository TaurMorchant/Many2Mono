# Поддержка прошлых LTS веток в монорепе

## Проблема

При объединении нескольких репозиториев в монорепу возникает вопрос: как обеспечить поддержку прошлых LTS (Long-Term Support) веток исходных репозиториев?

### Что нужно обеспечить

1. Возможность выпускать патчи для старых LTS версий
2. Согласованность версий зависимых модулей в рамках одной LTS
3. Минимизация сложности поддержки
4. Сохранение git-истории для LTS веток

---

## Контекст

### Текущая ситуация

- Каждый исходный репозиторий имеет свои LTS ветки с именованием `lts/25.3`, `lts/25.4` и т.д.
- Каждый модуль версионируется **независимо** (разные версии у разных репозиториев)
- Git-теги ставятся на каждую maven-версию модуля (не релизные теги, а теги версий)
- Модули **зависят друг от друга** — при патче одного модуля нужно обновлять зависимые
- За согласование версий отвечает **релизная тула** (самописная), которая:
  - Принимает список репозиториев для совместного релиза
  - Строит граф зависимостей
  - Повышает версии и обновляет зависимости
  - Делает релиз каждого репозитория
- **Ограничения релизной тулы:**
  - Не умеет отводить LTS ветки (делается вручную)
  - Не умеет работать с монорепой (только с отдельными репозиториями)

### Политика LTS

- 4 квартальных LTS в год (25.1, 25.2, 25.3, 25.4)
- Каждая LTS поддерживается 2 года
- В LTS ветках — только багфиксы

### Релизный процесс

- **Main ветка:** разработка → релиз изменённых модулей раз в спринт (на практике меняются все)
- **Конец квартала:** отводим LTS ветку от main
- **LTS ветка:** багфиксы по запросу

При любом релизе (спринтовый или багфикс):
1. Повысить версию изменённого модуля
2. Обновить версии в зависимых модулях (которые используют изменённый)
3. Это делает релизная тула

### Новые LTS (после миграции в монорепу)

Для новых LTS (26.2 и далее) процесс **однозначен**:

1. **Отведение LTS:** в конце квартала создаём ветку `lts/26.2` от main — обычная git-ветка, содержит все модули
2. **Багфиксы:** делаем в ветке `lts/26.2`, релизная тула обновляет версии изменённых и зависимых модулей
3. **Структура:** единая ветка со всеми модулями, как main

**Требуется:** адаптация релизной тулы для работы с монорепой (модули в одном репозитории вместо разных репозиториев).

**Вывод:** Для новых LTS проблем нет. Варианты решения (ниже) относятся только к **поддержке прошлых LTS** (25.3, 25.4 и т.д.), которые были отведены ещё в исходных репозиториях до миграции.

### Как работает Many2Mono сейчас

- Клонирует только `main` ветку каждого репозитория
- Применяет `git filter-repo` только к `refs/heads/main`
- Мержит истории с `--allow-unrelated-histories`
- LTS ветки **не переносятся** в монорепу

## Варианты решения

### Вариант 1: Полный перенос LTS веток (независимые истории)

**Идея:** LTS ветка собирается **отдельно** тем же процессом clone+filter-repo+merge. Получаются независимые истории без общего предка.

**Git-дерево:**
```
$ git log --graph --pretty=format:"%s%d" --all

*   Merge B  (HEAD -> main)
|\
| * B5
| * B4
| * B3
| * B2
| * B1
*   Merge A
|\
| * A5
| * A4
| * A3
| * A2
| * A1
* Init

*   Merge B  (lts/25.4)
|\
| * B-lts-1
| * B3
| * B2
| * B1
*   Merge A
|\
| * A-lts-1
| * A3
| * A2
| * A1
* Init (lts)
```

**Как создаётся:**
1. Для каждой ветки (main, lts/25.4) запускаем полный цикл:
   - `git clone --branch <branch>`
   - `git filter-repo --to-subdirectory-filter`
   - `git merge --allow-unrelated-histories`
2. Получаем независимые истории без общего предка

**Анализ:**

Два несвязанных дерева в git — **возможно**:
```bash
git checkout --orphan other-branch
git commit --allow-empty -m "Init"
# Создаёт новое дерево без связи с существующим
```

Плюсы:
- Единая LTS ветка — простой checkout
- Простой релизный процесс (всё в одной ветке)
- Низкая сложность CI/CD

Подводные камни:
1. **cherry-pick / rebase между ветками** — работает, но без общего предка больше конфликтов
2. **git log --all --graph** — показывает два отдельных дерева, сбивает с толку
3. **Дублирование истории** — коммиты A1-A3, B1-B3 существуют дважды (разные SHA), увеличивает размер репо
4. **Отсутствие связи** — `git log main..lts/25.4` бесполезен, `git diff` работает но без контекста

**Вывод:** Технически возможен. Простой в эксплуатации, но дублирует историю.

---

### Вариант 2: LTS ветки для каждого модуля в общем дереве

**Идея:** LTS ветки переносятся как часть общего дерева. Каждая LTS ветка отводится от "своего" коммита (A3 или B3) и содержит только файлы своего модуля.

**Git-дерево:**
```
$ git log --graph --pretty=format:"%s%d" --all

*   Merge B  (HEAD -> main)
|\
| * B5
| * B4
| | * B-lts-1  (lts/25.4/core-error-handling)
| |/
| * B3
| * B2
| * B1
*   Merge A
|\
| * A5
| * A4
| | * A-lts-1  (lts/25.4/core-utils)
| |/
| * A3
| * A2
| * A1
* Init
```

**Структура веток:**
- `main` — объединённая монорепа (core-utils/ + core-error-handling/)
- `lts/25.4/core-utils` — отведена от A3, содержит только файлы в core-utils/
- `lts/25.4/core-error-handling` — отведена от B3, содержит только файлы в core-error-handling/

**Как создаётся:**
1. Объединяем main ветки (как сейчас)
2. При клонировании каждого репозитория также клонируем его LTS ветки
3. После filter-repo LTS ветки остаются в дереве, отведённые от нужных коммитов
4. Имена веток переименовываются: `lts/25.4` → `lts/25.4/<module-name>`

**Как собирать релиз LTS 25.4?**

Нужен механизм, который:
1. Берёт `core-utils/` из ветки `lts/25.4/core-utils`
2. Берёт `core-error-handling/` из ветки `lts/25.4/core-error-handling`
3. Собирает их вместе для билда/релиза

Возможные реализации:
```bash
# CI/CD скрипт:
git checkout main  # или любая ветка с полной структурой
git checkout lts/25.4/core-utils -- core-utils/
git checkout lts/25.4/core-error-handling -- core-error-handling/
mvn deploy
```

**Анализ:**

Плюсы:
- LTS ветки — часть общего дерева (есть общий предок с main)
- `git log main..lts/25.4/core-utils` работает корректно
- cherry-pick между ветками проще (общая история)
- Нет дублирования коммитов A1-A3, B1-B3

Проблемы:
1. **Нет единой ветки LTS** — нельзя просто `git checkout lts/25.4` и получить всё
2. **Сложность CI/CD** — нужна специальная логика сборки
3. **Неполные ветки** — `git checkout lts/25.4/core-utils` даёт только core-utils/, остальных директорий нет
4. **Сложность релизного процесса** — при багфиксе модуля A нужно обновить зависимости в модуле C, но это разные ветки (`lts/25.4/core-utils` и `lts/25.4/core-error-handling`). Два коммита в двух ветках — как гарантировать атомарность? Как релизная тула будет с этим работать?
5. **Когерентность** — как гарантировать, что версии в разных ветках совместимы?

**Вывод:** Сохраняет связь с основным деревом, но усложняет релизный процесс и теряет единую точку сборки для LTS.

## Сравнение вариантов

| Аспект | Вариант 1 | Вариант 2 |
|--------|-----------|-----------|
| Единая LTS ветка | Да | Нет (по модулям) |
| Связь main и lts | Нет | Да |
| Простота checkout | Простой | Сложный |
| Сложность CI/CD | Низкая | Высокая |
| Сложность релиза в LTS | Низкая (одна ветка) | Высокая (координация веток) |
| Размер репо | Увеличен (дубли) | Нормальный |
| Теги версий | На модуль в единой ветке | На модуль в своей ветке |
| cherry-pick между ветками | Сложный (разные SHA) | Простой |

## Рассмотренные и отброшенные альтернативы

### Альтернатива A: Единая LTS ветка, отведённая от main монорепы

**Идея:** LTS ветка создаётся **после** объединения, как обычная ветка в монорепе.

**Почему не работает:**

```
*   Merge B  (HEAD -> main)
|\
| * B5
| * B4
| * B3    <-- если отвести LTS отсюда, нет файлов из A!
| * B2
| * B1
*   Merge A
|\
| * A5
| * A4
| * A3    <-- если отвести отсюда, нет файлов из B!
| * A2
| * A1
* Init    <-- пустой коммит, нет ничего
```

В монорепе **нет коммита**, где одновременно:
- Репо A в состоянии A3 (точка отведения LTS)
- Репо B в состоянии B3 (точка отведения LTS)

Единственный коммит с файлами из обоих репо — `Merge B`, но там уже A5 и B5.

**Вывод:** Невозможен в текущий условиях.

---

### Альтернатива B: LTS остаются в исходных репозиториях

**Идея:** Монорепа содержит только main, LTS поддержка продолжается в исходных репозиториях.

**Минусы:**
- Два места для поддержки кода (монорепа + исходные репо)
- Сложнее синхронизировать багфиксы
- Нарушает идею "единого источника правды"
- Со временем исходные репо устаревают и путаются

**Вывод:** Анти-паттерн для монорепы. **Не подходит** — цель отказаться от старых репо полностью.

---

### Альтернатива C: Гибридный подход — поэтапная миграция

**Идея:** Новые LTS создаются в монорепе, старые LTS остаются в исходных репозиториях до истечения срока поддержки.

**Минусы:**
- Временно поддерживаем два места
- Нужно чётко документировать, где какая LTS

**Вывод:** **Не подходит** — цель отказаться от старых репо полностью.

## Рекомендуемое решение

**Выбран Вариант 1: Полный перенос LTS веток (независимые истории)**

### Обоснование

Учитывая следующие факторы:
- **Временность проблемы:** Прошлые LTS (25.3, 25.4) живут максимум 2 года, после чего отомрут естественным образом
- **Критичность релизного процесса:** Багфиксы для клиентов требуют надёжного и простого процесса
- **Простота эксплуатации:** Единая LTS ветка упрощает работу команды (checkout, CI/CD, релиз)

Недостатки Варианта 1 (дублирование истории, увеличенный размер репо) являются **временными** и **некритичными** в сравнении с преимуществами.

### Что получаем

1. **Единая LTS ветка** для каждой версии (`lts/25.3`, `lts/25.4`)
2. **Простой checkout:** `git checkout lts/25.4` — и всё готово к работе
3. **Простой релизный процесс:** релизная тула работает как обычно (одна ветка, все модули внутри)
4. **Низкая сложность CI/CD:** стандартные пайплайны без специальной логики

### Что теряем

1. Связь между main и LTS ветками (разные деревья)
2. Дублирование истории до точки отведения LTS (~2x увеличение размера для старой истории)
3. Усложнённый cherry-pick между main и LTS (разные SHA, больше конфликтов)

### Вывод

Вариант 1 обеспечивает баланс между **простотой эксплуатации** и **временными техническими недостатками**, которые исчезнут через 2 года.

## План реализации

### Этап 1: Расширение Makefile для поддержки нескольких веток

**Цель:** Модифицировать Many2Mono для клонирования и объединения нескольких веток.

**Изменения в Makefile:**

1. Добавить параметр `BRANCHES` (по умолчанию `main`):
   ```makefile
   BRANCHES ?= main
   ```

2. Модифицировать `clone` target:
   ```bash
   clone: check-init
       @echo "==> Cloning repos and moving to subdirectories"
       mkdir -p "$(TMP_DIR)"

       while IFS='|' read -r url subdir || [[ -n "$$url" ]]; do
         [[ -z "$$url" || "$$url" == \#* ]] && continue
         repo="$$(basename "$$url" .git)"

         for branch in $(BRANCHES); do
           bare="$(TMP_DIR)/$${repo}-$${branch}.git"
           echo "---- Cloning $$repo ($$branch) -> $$subdir"

           rm -rf "$$bare"
           git clone --bare --branch $$branch --single-branch "$$url" "$$bare"

           (
             cd "$$bare"
             git filter-repo \
               --refs refs/heads/$$branch \
               --to-subdirectory-filter "$$subdir" \
               --force
           )
         done
       done < "$(REPOS_FILE)"
   ```

3. Модифицировать `merge` target:
   ```bash
   merge: check-init
       @echo "==> Creating monorepo from cloned repos"

       for branch in $(BRANCHES); do
         echo "==> Processing branch: $$branch"

         if [[ "$$branch" == "main" ]]; then
           rm -rf "$(MONOREPO_DIR)"
           git init -b main "$(MONOREPO_DIR)"
         else
           (cd "$(MONOREPO_DIR)" && git checkout --orphan $$branch)
         fi

         (
           cd "$(MONOREPO_DIR)"
           git commit --allow-empty -m "Initial monorepo commit ($$branch)"

           while IFS='|' read -r url subdir || [[ -n "$$url" ]]; do
             [[ -z "$$url" || "$$url" == \#* ]] && continue
             repo="$$(basename "$$url" .git)"
             bare="$(TMP_DIR)/$${repo}-$${branch}.git"

             if [[ ! -d "$$bare" ]]; then
               echo "[WARN] Skipping $$repo ($$branch) - not found" >&2
               continue
             fi

             git remote remove "$${subdir}-$${branch}" 2>/dev/null || true
             git remote add "$${subdir}-$${branch}" "$$bare"
             git fetch "$${subdir}-$${branch}" $$branch
             git merge "$${subdir}-$${branch}/$$branch" \
               --allow-unrelated-histories \
               -m "chore(monorepo): merge $$repo ($$branch) into /$$subdir"
           done < "$(REPOS_FILE)"
         )
       done
   ```

### Этап 2: Тестирование на тестовых репозиториях

1. Создать 2-3 тестовых репозитория с LTS ветками
2. Запустить `make all` с `BRANCHES="main lts/25.4"`
3. Проверить результат:
   - `git log --all --graph` показывает два дерева
   - `git checkout main` — все модули в актуальном состоянии
   - `git checkout lts/25.4` — все модули в LTS состоянии

### Этап 3: Миграция реальных репозиториев

**Подготовка:**
1. Определить список активных LTS веток (например: `lts/25.3`, `lts/25.4`)
2. Убедиться, что все репозитории имеют эти ветки (или решить, что делать, если нет)

**Запуск:**
```bash
make all BRANCHES="main lts/25.3 lts/25.4"
```

**Проверка:**
1. Проверить размер репозитория
2. Проверить структуру веток: `git branch -a`
3. Проверить историю: `git log --all --graph --oneline`
4. Проверить файлы в каждой ветке

### Этап 4: Адаптация релизной тулы

**Требования:**
1. Работа с модулями в одном репозитории вместо разных репозиториев
2. Поддержка нескольких веток (main, lts/*)
3. Построение графа зависимостей внутри монорепы
4. Обновление версий в pom.xml модулей

**Детали:** Требует отдельного дизайн-документа для релизной тулы.

### Этап 5: Настройка CI/CD

**Для каждой LTS ветки:**
1. Создать отдельный workflow/pipeline
2. Триггеры: push в `lts/*` ветки
3. Сборка и тесты всех модулей
4. Публикация артефактов

**Пример GitHub Actions:**
```yaml
name: LTS 25.4 CI

on:
  push:
    branches:
      - lts/25.4

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Set up JDK 17
        uses: actions/setup-java@v4
        with:
          java-version: '17'
      - name: Build with Maven
        run: mvn clean install
```

### Этап 6: Документирование

1. Обновить README монорепы с информацией о LTS ветках
2. Задокументировать процесс работы с LTS (checkout, багфиксы, релиз)
3. Обучить команду новому процессу
